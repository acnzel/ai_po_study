# 2강: Claude를 처음 만나는 법
## — 기초 프롬프팅

> **소요시간**: 45분  
> **이 강의를 마치면**: 5가지 프롬프팅 패턴을 사용해서 PRD 섹션 하나를 완성할 수 있다

---

## 프롬프트가 뭔가요?

**프롬프트(Prompt)** = Claude에게 보내는 메시지

좋은 프롬프트를 쓰면 좋은 결과가 나오고, 나쁜 프롬프트를 쓰면 나쁜 결과가 나옵니다.  
하지만 걱정하지 마세요. 패턴 5가지만 익히면 됩니다.

---

## 핵심 원칙: 한 번에 다 하려 하지 말 것

AI를 쓰면서 가장 먼저 깨달은 게 하나 있어요.
원하는 걸 한 번에 다 넣으려 할수록 결과는 오히려 더 엉망이 됩니다.
처음부터 완벽한 프롬프트를 작성하려고 하지 마세요.

### 왜 그럴까요?

사람도 마찬가지입니다. 처음 만난 동료에게 "우리 프로젝트의 전체 기획서를 써줘"라고 하면 어떻게 될까요?

도메인 지식도 없고, 어떤 순서로 일 하는지도 모르고, 우리 팀의 특성도 모르는 상태에서는 아무리 능력 있는 사람도 좋은 결과를 낼 수 없습니다.

Claude도 똑같아요. **결과물을 만들기 이전에 만들고자하는 것에 대해 설명을 충분히 하면, 훨씬 좋은 결과가 나옵니다.**

---

## 나쁜 예시 vs 좋은 예시

### ❌ 한 방에 다 요구하는 방식

```
"우리 앱의 알림 설정 기능 PRD를 작성해줘요
사용자 분석은 MZ세대 중심이고, 기술 스택은 React Native고, 
출시일은 3월이야. 알림 종류는 마케팅 알림, 매치 관련 알림, 
시스템 알림 3가지가 있고, 각각 설정을 끄고 켤 수 있어야 합니다.
엣지케이스도 포함하고 UX 흐름도 고려해서 써줘요. 지금 바로 완성본으로."
```

결과: 우리 서비스를 전혀 모르는 채로 작성된, 수정이 엄청 필요한 PRD

---

### ✅ 단계적으로 쌓아가는 방식

```
1단계: "우리 앱에 대해 설명하겠습니다. [앱 설명 or 회사 사이트나 문서 링크도 좋음] — 이해한 내용 요약해줘요."

2단계: "알림 설정 기능을 추가하려 합니다. 어떤 것들을 고려해야 할까요?"

3단계: "좋아요. 이제 이 기능의 PRD 초안을 작성해줘요."

4단계: "작성한 PRD를 개발자 관점에서 비판적으로 검토해줘요."
```

결과: 맥락을 이해한 채 작성된, 바로 팀에 공유할 수 있는 PRD

---

## 실전 프롬프팅 패턴 5가지

### 패턴 1: 탐색 먼저

> 작업을 시키기 전에, 먼저 상황을 파악하게 하는 패턴

**언제 쓰나요?**  
새로운 기능을 기획할 때, 어디서부터 시작해야 할지 막막할 때

**기본 형태:**
```
[주제]에 대해 탐색해주세요.
어떤 내용을 다루는지 요약하고,
내가 고려해야 할 핵심 요소들을 정리해주세요.
```

**실전 예시:**
```
매치 필터 기능 구현에 대해 탐색해줘요.
어떤 종류의 필터가 있는지,
사용자 입장에서 직관적이고 쉽게 사용되려면 어떤 UX와 기능이 필요한지,
내가 기획할 때 놓치기 쉬운 요소들이 뭐가 있는지 정리해주세요.
```

---

### 패턴 2: 구조 제안 요청

> 내용을 쓰기 전에, 먼저 구조만 잡아보는 패턴

**언제 쓰나요?**  
문서를 쓰는데 어떻게 구성해야 할지 모를 때

**기본 형태:**
```
[문서 유형]을 작성할 때 담아야 할 섹션과 순서를 제안해줘요
직접 쓰지는 말고, 구조만 먼저 잡아서 보여줘요
Claude가 가끔 먼저 달려나가는 경험들을 해보셨을 겁니다. 워워 진정 시키고 내 말을 끝까지 듣게 하는 연습이 필요해요.
```

**실전 예시:**
```
소셜 로그인 기능 PRD를 작성할 때 담아야 할 섹션과 순서를 제안해줘요
직접 내용을 쓰지는 말고, 어떤 항목들이 필요한지 구조만 제안해줘요
각 섹션이 왜 필요한지도 한 줄씩 설명해줘요
```

---

### 패턴 3: 비판적 검토

> 작성된 내용을 특정 관점에서 비판하게 하는 패턴

**언제 쓰나요?**  
초안을 썼는데 뭔가 빠진 것 같을 때, 팀에 공유하기 전에

**기본 형태:**
```
방금 작성한 내용을 [역할] 관점에서 비판적으로 검토해주세요.
무엇이 빠졌고, 무엇이 불명확한지 알려주세요.
```

**실전 예시:**
```
방금 작성한 결제 실패 알림 PRD를 백엔드 개발자 관점에서 비판적으로 검토해주세요.

구현하기 전에 반드시 결정해야 하는 것들, 이 PRD만 봐서는 개발을 시작할 수 없는 불명확한 부분, 기술적으로 어려울 수 있는 부분을 찾아주세요.
```

---

### 패턴 4: 반복 개선

> 피드백을 반영해서 버전을 올려가는 패턴

**언제 쓰나요?**  
검토 후 개선이 필요할 때

**기본 형태:**
```
위 피드백을 반영해서 개선된 버전을 만들어주세요.
변경한 부분과 이유도 함께 설명해주세요.
```

**실전 예시:**
```
방금 검토해준 내용을 반영해서 PRD를 개선해주세요.

특히 다음 두 가지 보완이 필요합니다:
1. 알림 발송 실패 케이스에 대한 처리 방안
2. 사용자가 모든 알림을 끄는 경우 운영자에게 알려야 하는지 여부

변경한 섹션마다 어떤 내용을 추가했는지 표시해주세요.
```

---

### 패턴 5: 관점 전환

> 같은 내용을 다른 독자에게 맞게 재작성하는 패턴

**언제 쓰나요?**  
같은 기획 내용을 CEO, 개발팀, 디자이너 등에게 다르게 전달해야 할 때

**기본 형태:**
```
같은 내용을 개발자가 읽는다고 가정하고 다시 써주세요.
```

**실전 예시:**
```
이 PRD의 핵심 내용을 CEO에게 보고하는 형식으로 다시 써주세요.

CEO는 기술 세부사항보다는 다음이 궁금할 거라 이 내용이 반드시 추가 되어야 해:
- 이 기능이 왜 필요한가 (비즈니스 임팩트)
- 얼마나 걸리나 (일정)
- 성공을 어떻게 측정할 건가 (지표)

1페이지 분량으로 핵심만 담기도록 재구성 해주세요.
```

---

## 패턴을 조합하면 더 강력해집니다

실제 업무에서는 패턴을 순서대로 조합해서 씁니다:

```
[패턴 1: 탐색]
"소셜 로그인 기능 추가에 대해 탐색해주세요.
 고려해야 할 요소들을 정리해주세요."

↓ Claude 답변 확인

[패턴 2: 구조 제안]
"이 내용을 바탕으로 PRD 구조를 잡아주세요."

↓ 구조 확인 및 수정

[패턴 4: 초안 작성]
"이 구조대로 PRD 초안을 작성해주세요."

↓ 초안 확인

[패턴 3: 비판적 검토]
"개발자 관점에서 이 PRD를 검토해주세요."

↓ 검토 내용 확인

[패턴 4: 반복 개선]
"검토 결과를 반영해서 개선해주세요."

↓ 완성본 확인

[패턴 5: 관점 전환 (필요시)]
"이 내용을 CEO 보고용으로 요약해주세요."
```

## 실용 팁

### 팁 1: 구체적일수록 좋다

❌ "잘 써줘"  
✅ "10년 차 개발자도 이해할 수 있도록 기술적 세부사항을 명확히 써주세요"

❌ "더 좋게 해줘"  
✅ "엣지케이스 섹션에 '로그인 중 앱이 종료되는 경우'와 '소셜 계정이 탈퇴된 경우'를 추가해주세요"

### 팁 2: 역할을 지정하면 관점이 달라진다

```
"[시니어 UX 디자이너] 관점에서 이 화면 플로우를 검토해주세요."
"[사용자 중 가장 IT에 익숙하지 않은 60대 사용자] 입장에서 이 기능을 평가해주세요."
"[스타트업 CTO] 관점에서 이 기술적 결정의 리스크를 알려주세요."
```

### 팁 3: 형식을 지정하면 쓰기 편하다

```
"불릿 포인트로 정리해주세요."
"표 형태로 비교해주세요."
"슬랙에 바로 붙여넣을 수 있는 형식으로 써주세요."
"A4 한 장 분량으로 요약해주세요."
```

### 팁 4: 모르면 물어보라고 해라

```
"PRD를 작성하기 전에, 결과물의 품질을 높이기 위해
 나에게 먼저 물어봐야 할 것들이 있으면 질문해주세요."
```

이렇게 하면 Claude가 먼저 필요한 정보를 요청합니다. 그 질문들을 보면 내가 뭘 빠뜨리고 있는지도 알 수 있어요.

## ✏️ 오늘의 실습 과제

**목표**: 5가지 패턴을 각각 한 번씩 써보며 PRD 섹션 하나 완성하기

**따라하기**:

아래 시나리오를 사용하거나, 자신의 실제 업무로 바꿔서 해보세요.

> **시나리오**: 커머스 앱의 "최근 본 상품" 기능 기획

**Step 1 — 패턴 1 (탐색)**
```
커머스 앱에서 '최근 본 상품' 기능을 기획하려 합니다.
이 기능의 일반적인 구성 방식, 사용자가 기대하는 동작,
주요 커머스 앱들의 구현 방식을 탐색해주세요.
내가 놓치기 쉬운 기획 포인트도 알려주세요.
```

**Step 2 — 패턴 2 (구조)**
```
이 기능의 PRD에 들어가야 할 섹션 구조를 제안해주세요.
내용은 아직 쓰지 말고 구조만.
```

**Step 3 — 초안 작성 요청**
```
이 구조로 PRD 초안을 작성해줘.
우리 앱은 30대 여성이 주요 사용자인 패션 커머스야.
```

**Step 4 — 패턴 3 (비판)**
```
작성한 PRD를 앱 개발자와 UX 디자이너 관점에서 검토해주세요.
구현하기 전 반드시 결정해야 할 것들을 찾아주세요.
```

**Step 5 — 패턴 4 (개선)**
```
검토 내용을 반영해서 엣지케이스 섹션을 보완해주세요.
```

**기록할 것**: 각 단계에서 Claude의 답변이 어떻게 달라졌는지 메모해두세요.
