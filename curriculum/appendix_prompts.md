# 부록: 복사해서 쓰는 프롬프트 모음

> **사용법**: `[대괄호]` 안의 내용을 자신의 상황에 맞게 바꿔서 사용하세요.  
> 대괄호를 그대로 두면 Claude가 내용을 채워달라고 요청할 수 있습니다.

---

## 📋 PRD 작성 프롬프트

### 탐색 시작

```
[기능명] 기능을 기획하려 합니다.

이 기능에 대해 탐색해줘:
1. 이 기능이 해결하는 핵심 사용자 문제
2. 유사 서비스들의 구현 방식 (알고 있는 경우)
3. 기획 시 반드시 챙겨야 하는데 놓치기 쉬운 케이스들
4. 개발팀이 가장 많이 질문할 것 같은 부분

우리 서비스: [서비스 간략 설명]
주요 사용자: [사용자 설명]
```

---

### PRD 구조 먼저 잡기

```
[기능명] PRD에 들어가야 할 섹션 구조를 제안해줘.
내용은 아직 쓰지 말고, 어떤 항목이 필요한지 구조만.
각 섹션이 왜 필요한지 한 줄씩 이유도 붙여줘.
```

---

### PRD 초안 작성

```
아래 정보로 PRD 초안을 작성해줘.

기능명: [기능명]
해결하려는 문제: [문제 설명]
핵심 사용자: [사용자 설명]
반드시 포함할 것: [있으면 명시]
이번 범위에서 제외할 것: [있으면 명시]
팀 제약: [개발 n명, n주 일정 등]

섹션 구조:
- 요약 (3줄 이내)
- 배경 및 목적
- 목표 사용자 & 시나리오
- 기능 요구사항 (Must Have / Nice to Have / Out of Scope)
- UX 흐름
- 엣지케이스 & 예외 처리
- 성공 지표
- Open Questions
```

---

### 개발자 관점 검토

```
위 PRD를 시니어 백엔드 개발자 관점에서 검토해줘.

찾아줄 것:
1. 구현 전 PO에게 반드시 확인해야 할 사항들
2. 이 PRD만 보고는 개발을 시작할 수 없는 불명확한 부분
3. 데이터 처리에서 빠진 케이스 (생성/수정/삭제/오류)
4. 기술적으로 예상보다 복잡할 수 있는 요구사항
```

---

### UX 디자이너 관점 검토

```
위 PRD를 시니어 UX 디자이너 관점에서 검토해줘.

찾아줄 것:
1. 사용자가 혼란스러워할 화면 또는 플로우
2. UX 흐름에서 빠진 화면 상태 (로딩, 에러, 빈 화면 등)
3. 사용자가 실수하기 쉬운 지점과 방지 방법
4. 접근성 측면에서 고려해야 할 것들
```

---

### 사용자 관점 검토

```
위 PRD를 실제 사용자 관점에서 평가해줘.
사용자 프로필: [나이/직업/사용 패턴 등]

평가할 것:
1. 이 기능을 쓰다가 막힐 것 같은 순간
2. 기대했는데 없어서 실망할 기능
3. 있긴 한데 실제로 쓸 것 같지 않은 기능
4. "왜 이게 이렇게 동작하지?"라고 느낄 순간들
```

---

### 혹독한 자기 비판

```
방금 작성한 PRD에서 가장 큰 문제점 3가지를 찾아줘.
솔직하게, 혹독하게 비판해줘.
"이 PRD를 팀에 공유하면 안 되는 이유"를 찾는다고 생각하고.
```

---

### 엣지케이스 강화

```
이 PRD의 엣지케이스 섹션을 보강해줘.

다음 측면에서 빠진 케이스를 찾아줘:
- 네트워크 오류 상황
- 권한 없는 사용자 접근
- 데이터가 없거나 불완전한 경우 (empty state)
- 동시에 여러 사용자가 같은 데이터를 수정하는 경우
- 앱 업데이트 중간에 사용하는 경우
- 사용자가 연속으로 같은 액션을 빠르게 하는 경우
```

---

### 최종 완성

```
검토 내용을 반영해서 PRD를 최종 버전으로 업데이트해줘.

업데이트 시:
1. Open Questions에서 답변 가능한 것은 채워줘
2. 각 Must Have 항목이 "개발자가 바로 작업 가능한 수준"인지 확인
3. 요약 섹션이 2문단 이내에 핵심을 담고 있는지 확인
4. 변경한 내용을 마지막에 변경 로그로 정리해줘
```

## 📅 회의 관련 프롬프트

### 회의 아젠다 준비

```
[회의 목적]을 위한 [n분] 회의 아젠다를 만들어줘.

참석자: [역할 목록]
결정해야 할 것들: [결정 사항 목록]

각 아젠다 항목마다:
- 소요 시간
- 논의 포인트
- 목표 결과물 (무엇을 결정하고 나가야 하는가)
```

---

### 회의록 작성

```
아래 회의 내용을 회의록으로 정리해줘.

[회의 내용 붙여넣기]

형식:
- 회의 일시 & 참석자
- 논의 내용 요약 (불릿 포인트)
- 결정 사항 (번호 목록, 명확하게)
- 액션 아이템 (담당자 / 기한 포함)
- 다음 회의까지 확인해야 할 것들
```

---

### 이해관계자 보고 요약

```
아래 내용을 [CEO / CPO / 개발리드]에게 보고하는 형식으로 정리해줘.

[원본 내용]

[CEO]는 이것에 관심:
- 비즈니스 임팩트 (숫자로)
- 일정 (언제 출시)
- 리스크 (뭐가 문제될 수 있나)

[CPO]는 이것에 관심:
- 사용자 가치
- 전략적 방향성
- 우선순위 근거

[개발리드]는 이것에 관심:
- 기술적 결정 사항
- 일정 실현 가능성
- 팀 리소스 영향

해당하는 역할에 맞게 핵심만 [n]페이지로 요약해줘.
```

---

### 스프린트 계획 미팅 준비

```
다음 스프린트에 [기능명]을 개발할 예정이야.
내일 스프린트 계획 미팅 준비를 도와줘.

현재 상황:
- 팀: [개발 n명, 디자인 n명]
- 스프린트 기간: [n주]
- 이미 확정된 작업: [있으면 명시]

준비해줄 것:
1. 계획 미팅 아젠다 (30분 기준)
2. 개발리드에게 확인해야 할 기술적 질문 5개
3. 이번 스프린트 목표 한 줄 정의
```

## 📊 분석 관련 프롬프트

### 기능 요청 분석

```
아래 기능 요청을 분석해줘.

요청 내용: [원문 그대로 붙여넣기]
요청 출처: [CS 채널 / 사용자 인터뷰 / 내부 요청 등]

분석:
1. 사용자가 진짜 원하는 것 (표면 요청 vs 근본 니즈)
2. 이 요청이 도움되는 사용자 비율 예상
3. 기존 기능으로 해결 가능한지 여부
4. 구현 복잡도 예상 (Low / Medium / High)
5. 우선순위 제안 (High / Medium / Low) + 근거
6. 다음 액션 (PO가 무엇을 해야 하는가)
```

---

### 사용자 피드백 분류

```
아래 사용자 피드백들을 분류하고 분석해줘.

[피드백 목록 붙여넣기]

분류 기준:
- 버그 신고
- 기능 요청
- UX/UI 불편
- 성능 문제
- 긍정 피드백
- 기타

분석 결과:
1. 카테고리별 건수
2. 가장 자주 언급된 키워드 Top 5
3. 즉시 조치 필요한 이슈 (있다면)
4. 다음 스프린트에서 고려할 개선점 3가지
```

---

### 경쟁사 기능 비교

```
[기능명]에 대해 [경쟁사 A]와 [경쟁사 B]를 비교 분석해줘.

비교 항목:
- 기본 기능 구성
- UX 차별점
- 우리가 배울 수 있는 점
- 우리가 차별화할 수 있는 기회

우리 서비스: [설명]
우리의 강점: [설명]
```

---

### 데이터 인사이트 요약

```
아래 데이터를 비개발자도 이해할 수 있게 요약해줘.

[데이터 붙여넣기]

포함할 것:
1. 핵심 수치 3가지 (가장 중요한 것)
2. 이 데이터가 말해주는 것 (인사이트)
3. 우리가 취해야 할 액션
4. 추가로 확인이 필요한 것들

형식: 슬랙에 바로 공유 가능한 형태
```

## 🔧 Soul Document & 시스템 관련 프롬프트

### Soul Document 만들기

```
나는 [회사명]의 PO입니다.
앞으로 나와 Claude가 잘 협업하기 위한 Soul Document(CLAUDE.md)를 만들고 싶어요.

작성하기 전에 필요한 정보를 질문해주세요.
최대 6가지 질문으로 핵심만.
```

---

### Soul Document 업데이트

```
기존 CLAUDE.md를 업데이트하려 해.

[기존 내용 붙여넣기]

추가/수정할 내용:
- [새로운 결정 또는 컨텍스트]

위 내용을 반영해서 업데이트된 버전을 만들어줘.
변경한 부분을 표시해줘.
```

---

### Command 만들기

```
내가 자주 하는 [업무 설명] 작업을 Command로 만들고 싶어.

이 작업에 대해:
- 필요한 입력값은 무엇인가?
- 출력 형식은 어떻게 하면 좋은가?
- 어떤 조건들을 항상 지켜야 하는가?

위 내용을 바탕으로 Command 명세서를 작성해줘.
그리고 실제 작동하는 프롬프트도 포함해줘.
```

---

### Command 개선

```
내가 쓰는 Command야:
[Command 내용 붙여넣기]

이 Command를 사용했을 때 아쉬운 점:
[아쉬운 점 설명]

어떻게 개선하면 좋을지 제안해줘.
개선된 버전의 프롬프트도 함께 작성해줘.
```

```
[원본 내용]
```

### 의사결정 지원

```
다음 두 가지 옵션 중 어떤 것이 더 나은지 판단하는 데 도움줘.

옵션 A: [설명]
옵션 B: [설명]

판단 기준:
- 사용자 경험
- 개발 비용
- 비즈니스 임팩트

각 옵션의 장단점을 정리하고, 어떤 상황에서 어떤 걸 선택하면 좋을지 알려줘.
```
