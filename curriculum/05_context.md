# 5강: 컨텍스트 쌓기
## — 질문 먼저, 작업 나중에

> **소요시간**: 45분  
> **이 강의를 마치면**: 작업 전 대화로 컨텍스트를 쌓는 전략을 실무에 적용할 수 있다

---

## "30분을 먼저 쓰면 나머지 시간이 10배 빨라진다"

신정규 대표의 개발 과정을 보면 흥미로운 패턴이 있습니다.

실제 코드 작성을 시작하기 전에, **먼저 Claude와 대화로 컨텍스트를 충분히 쌓습니다.**

> *"내가 뭘 하고 싶은지가 명확하지만,  
>  그건 내 머릿속에만 명확하기 때문에  
>  Claude의 컨텍스트에 집어넣기 위해서  
>  사전 조사를 시키는 거예요."*  
> — 신정규 (Lablup 대표)

PO 입장에서도 똑같이 적용됩니다.

---

## 컨텍스트란 무엇인가?

**컨텍스트(Context)** = 현재 대화에서 Claude가 알고 있는 배경 정보

같은 질문이라도 컨텍스트가 얼마나 쌓였느냐에 따라 결과가 완전히 달라집니다.

### 컨텍스트가 없을 때

```
나: "알림 기능 PRD 써줘"
Claude: (일반적인 알림 기능 PRD 작성)
→ 우리 서비스가 뭔지 모름
→ 사용자가 누구인지 모름
→ 왜 이 기능이 필요한지 모름
```

### 컨텍스트가 쌓였을 때

```
[20분간 대화 후]
나: "알림 기능 PRD 써줘"
Claude: (30대 여성 패션 커머스 앱임을 알고,
         Q1 전환율 목표와 연결해서,
         우리 팀 개발 방식에 맞는 수준으로 작성)
```

---

## 컨텍스트 쌓기 3단계

### Step 1: 배경 조사 시키기

> "이 주제가 어떻게 생겼는지, Claude가 먼저 탐색하게 만들기"

```
[기능/주제]에 대해 탐색해줘.

우리 상황에서 특히 중요하게 고려해야 할 것들이 뭐가 있는지,
일반적으로 알려진 Best Practice가 있는지,
그리고 내가 미처 생각 못한 관점이 있는지 알려줘.
```

왜 이렇게 하나요? Claude가 주제를 충분히 탐색하면서, **동시에 내가 미처 생각 못했던 부분을 발견**하게 됩니다.

---

### Step 2: 제약 조건 명확화

> "우리 상황에 맞는 제약들을 명확히 하기"

```
우리 상황을 고려했을 때:
- 팀 규모: [개발 n명, 디자인 n명]
- 출시 일정: [n주 후]
- 기술적 제약: [있으면 명시]

이 상황에서 반드시 챙겨야 할 것과,
현실적으로 타협해야 할 것을 구분해줘.
```

왜 이렇게 하나요? **이상적인 기획** vs **현실적인 기획**의 차이를 명확히 하기 위해서입니다.

---

### Step 3: 작업 실행

> "컨텍스트가 충분히 쌓인 상태에서 실제 작업 요청"

```
위에서 논의한 내용을 바탕으로 [문서명]을 작성해줘.
```

---

## 실전 예시: 스프린트 계획 미팅 준비

### 상황

다음 스프린트에서 "소셜 로그인" 기능을 출시해야 합니다. 내일 스프린트 계획 미팅이 있는데, 30분 준비를 어떻게 해야 할지 모릅니다.

---

#### Step 1: 배경 조사

```
나: 모바일 앱에서 소셜 로그인 기능을 다음 스프린트에 출시할 예정이야.
    이 기능 출시 전 PO가 확인해야 할 것들을 정리해줘.
    
    특히:
    - 개발팀이 반드시 결정받아야 할 것들
    - QA에서 자주 빠뜨리는 테스트 케이스
    - 출시 후 바로 확인해야 할 지표들

Claude: [탐색 결과를 정리해서 답변]
        소셜 로그인 출시 전 확인사항:
        
        개발팀이 결정받아야 할 것들:
        1. 연동할 소셜 플랫폼 (카카오, 네이버, 애플, 구글)
        2. 기존 이메일 계정과 소셜 계정 연결 허용 여부
        3. 소셜 계정 탈퇴 시 서비스 계정 처리 방식
        ... (계속)
```

---

#### Step 2: 제약 조건 명확화

```
나: 우리 상황이야:
    - 개발팀 4명, 2주 스프린트
    - 이미 다른 기능 작업이 있어서 소셜 로그인에 쓸 수 있는 시간은 
      개발자 1명 기준 1주일
    - 이번에 반드시 출시해야 하는 이유: 가입 전환율 낮음
    
    이 상황에서 이번 스프린트에서 할 것과, 
    다음으로 미룰 것을 나눠줘.

Claude: [현실적인 범위 제안]
        이번 스프린트 (Must Do):
        - 카카오 로그인 1개만 (가장 사용률 높음)
        - 신규 가입만 (기존 계정 연결은 다음으로)
        
        다음 스프린트 (Defer):
        - 네이버, 구글, 애플 추가
        - 기존 이메일-소셜 계정 연결
```

---

#### Step 3: 미팅 준비 자료 생성

```
나: 이 내용을 바탕으로 내일 30분짜리 스프린트 계획 미팅 
    아젠다와 논의 포인트를 만들어줘.
    
    참석자: PO(나), 개발리드, 디자이너 1명
    목표: 소셜 로그인 개발 시작을 위한 의사결정 완료

Claude: [미팅 아젠다 작성]
        ## 소셜 로그인 스프린트 계획 미팅 아젠다
        
        총 30분 | 참석: PO, 개발리드, 디자이너
        
        [10분] 이번 스프린트 범위 확인
        - 카카오만 우선 출시 결정 배경 공유
        - 개발리드 피드백 받기
        
        [10분] 의사결정 필요 사항
        - 기존 회원과 소셜 계정 연결: 이번에? 다음에?
        - 소셜 계정 탈퇴 시 처리: 정책 확정 필요
        
        [10분] 일정 및 다음 스텝
        - 개발 시작 일자 확인
        - 디자인 필요 사항 리스트업
        
        결정하고 나가야 할 것:
        ☐ 이번 스프린트 최종 범위
        ☐ 탈퇴 처리 정책
        ☐ QA 시작 일자
```

**30분 준비로 미팅이 훨씬 구체적이고 효율적**해집니다.

## PROGRESS.md — 진행 상황 추적 파일

> 대화가 길어지거나 다음 날 이어서 작업할 때 매우 유용합니다

Claude는 새 대화를 시작하면 이전 내용을 기억하지 못합니다.  
**PROGRESS.md에 중간 상황을 기록해두면**, 새 대화에서도 빠르게 이어받을 수 있습니다.

### PROGRESS.md 작성법

```markdown
# [기능명] PRD 작성 현황

## 목표
- 목적: [무엇을 만들려고 하는가]
- 출시 목표일: [날짜]
- 담당 개발자: [이름]

## 진행 상황
- [x] Phase 1: 탐색 완료 (2026-02-24)
- [x] Phase 2: 구조 확정 (2026-02-24)
- [ ] Phase 3: 초안 작성 (진행 중)
- [ ] Phase 4: 개발자 검토
- [ ] Phase 5: 최종 완성

## 확인된 주요 내용
- 결제 실패 유형: 3가지 (일시 오류 / 카드 문제 / 사용자 취소)
- 알림 발송 타이밍: 실패 즉시 + 24시간 후 1회
- 카드사 오류 코드: 자체 분류 사용 (백엔드팀 기존 코드 활용)

## Open Questions (미결)
1. 재시도 알림을 푸시로만 할지, 카카오톡도 추가할지
2. 결제 실패 로그 보관 기간

## 다음 작업 시 이어받기용 프롬프트
"결제 실패 재시도 기능 PRD 작성 중입니다. 
 Phase 3 초안 작성이 남아있어요. 
 위 PROGRESS.md 내용을 읽고 이어서 도와주세요."
```

### PROGRESS.md 활용법

새 대화에서 이어받을 때:

```
[PROGRESS.md 내용 붙여넣기]

위 내용을 바탕으로 이어서 작업해주세요.
Phase 3 초안 작성이 필요합니다.
```

## 컨텍스트가 쌓이면 달라지는 것

| | 컨텍스트 없음 | 컨텍스트 풍부 |
|--|--------------|--------------|
| PRD 내용 | 일반적인 기능 설명 | 우리 서비스 맥락에 맞는 내용 |
| 해결책 | 교과서적 Best Practice | 우리 팀 규모/일정에 맞는 현실적 방안 |
| 엣지케이스 | 보편적인 케이스 | 우리 사용자가 실제 겪을 케이스 |
| 우선순위 | 일반적 우선순위 | 우리 Q1 목표에 맞춘 우선순위 |
| 용어 | 업계 일반 용어 | 우리 팀 내부 용어 사용 |

---

## 자주 하는 실수: "이 대화 저장하고 싶은데"

Claude 대화에서 정말 중요한 내용이 나왔을 때, 그냥 지나치면 나중에 못 찾습니다.

**중요한 내용이 나오면 즉시 기록하세요:**

```
나: "방금 정리해준 결제 실패 유형 분류와 처리 방식을 
     나중에 참고할 수 있도록 마크다운 형식으로 정리해줘.
     PROGRESS.md에 추가할 수 있도록."
```

## ✏️ 오늘의 실습 과제

**목표**: 20분 컨텍스트 쌓기 대화 후 PROGRESS.md 작성

**따라하기**:

1. 현재 기획 중인 프로젝트 하나 선택 (없으면 4강 시나리오 계속 사용)

2. 아래 순서로 대화 진행:
   ```
   Step 1 (7분): 배경 조사 탐색 프롬프트로 대화
   Step 2 (7분): 제약 조건 명확화
   Step 3 (6분): 미팅/문서 준비 요청
   ```

3. 대화 후 PROGRESS.md 작성:
   - `templates/PROGRESS_md_template.md` 파일 사용
   - 확인된 주요 내용 정리
   - 미결 사항 기록

4. 다음 날 새 대화에서 PROGRESS.md를 활용해서 이어서 작업해보기
