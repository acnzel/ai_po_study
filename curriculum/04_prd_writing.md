# 4강: PRD를 Claude와 함께 쓰는 법
## — PRD 작성 실전

> **소요시간**: 60분  
> **이 강의를 마치면**: 실제 기능의 PRD를 40분 안에 완성할 수 있다

---

## PRD 작성, 왜 항상 시간이 오래 걸릴까?

PRD 작성이 오래 걸리는 진짜 이유들:

1. **어디서 시작해야 할지 막막하다** — 빈 화면 앞에서 멈춤
2. **한 번에 완벽하게 쓰려 한다** — 첫 문장부터 막힘
3. **검토 없이 공유했다가 망신** — 개발팀에서 "이럴 때 어떻게 해요? 저럴 때 어떻게 해요?" 질문 폭격
4. **엣지케이스를 개발 도중에 발견** — 일정 지연, 재기획

Claude를 제대로 활용하면 이 4가지 문제를 모두 해결할 수 있습니다.

---

## PRD 작성 5단계 플로우

전체 과정을 **40분 안에** 마칠 수 있습니다.

```
Phase 1: 탐색    (10분) ─ 뭘 만들어야 하는지 파악
Phase 2: 구조    (5분)  ─ 어떤 섹션이 필요한지 확정
Phase 3: 초안    (10분) ─ 내용 채우기
Phase 4: 검토    (10분) ─ 다각도로 비판 받기
Phase 5: 완성    (5분)  ─ 팀 공유용 최종 버전
```

---

## Phase 1: 탐색 (10분)

> **목적**: 기능의 배경, 사용자 니즈, 고려해야 할 요소 파악

### 왜 이 단계가 중요한가?

PO가 기획을 망치는 가장 흔한 이유: **처음 생각한 방향이 사실 사용자 니즈와 달랐다.**

탐색 단계에서 Claude와 대화하다 보면, "아, 이건 내가 못 보고 있었네"를 발견하게 됩니다.

### 탐색 프롬프트 (복사해서 쓰세요)

```
[기능명] 기능을 기획하려 합니다.

이 기능에 대해 탐색해줘:
1. 이 기능이 해결하는 사용자 문제는 무엇인가?
2. 유사 서비스들은 어떻게 구현했나? (알고 있는 경우)
3. 기획 시 놓치기 쉬운 케이스들이 뭐가 있나?
4. 이 기능 구현 시 개발팀이 꼭 알아야 할 것들은?

우리 서비스 정보: [Soul Document에서 핵심 내용 붙여넣기]
```

### 탐색 후 할 일

Claude 답변을 읽으면서:
- ✅ 맞는 내용은 메모
- ✏️ 우리 상황과 다른 내용은 수정 메모
- 💡 내가 놓쳤던 내용은 추가 메모
- 이 메모들을 md 파일들에 업데이트 해주세요 (Claude에게 전달해서 다음번 작업 시 참고 될 수 있도록 내용 반영 하기)

---

## Phase 2: 구조 설계 (5분)

> **목적**: PRD에 어떤 섹션이 필요한지 확정

### 기본 PRD 구조 (PO 표준)

아래 구조가 대부분의 기능에 적합합니다. 기능 특성에 따라 섹션을 추가하거나 생략하세요.

```markdown
## [기능명] PRD

### 0. 요약 (Summary)
한 문단으로 이 기능이 무엇인지, 왜 만드는지

### 1. 배경 및 목적
- 이 기능을 만드는 이유
- 해결하려는 사용자 문제
- 기대하는 비즈니스 효과

### 2. 목표 사용자
- 주요 사용자 페르소나
- 이 기능을 가장 많이 쓸 사람

### 3. 사용자 시나리오 (As-Is → To-Be)
- 현재 상황 (기능 없을 때)
- 이 기능이 있을 때

### 4. 기능 요구사항
- Must Have (없으면 출시 불가)
- Nice to Have (있으면 좋은 것)
- Out of Scope (이번엔 안 만들 것)

### 5. UX 흐름
- 핵심 사용자 플로우 (화면 순서)
- 각 화면에서 일어나는 일

### 6. 엣지케이스 & 예외 처리
- 예외 상황 목록
- 각 예외에 대한 처리 방안

### 7. 성공 지표
- 이 기능의 성공을 어떻게 측정하는가?
- 출시 후 n주 내 확인할 지표

### 8. Open Questions
- 아직 결정 안 된 것들
- 개발팀/디자인팀과 논의 필요한 것들
```

---

## Phase 3: 초안 작성 (10분)

> **목적**: 각 섹션의 내용을 채우기

### 초안 작성 프롬프트 (복사해서 쓰세요)

```
Phase 1에서 탐색한 내용과 아래 PRD 구조를 바탕으로 
[기능명] PRD 초안을 작성해줘.

[PRD 구조 붙여넣기]

추가 컨텍스트:
- 우리 서비스: [서비스 설명]
- 주요 사용자: [사용자 설명]
- 이 기능의 핵심 가치: [한 줄 설명]
- 반드시 포함해야 할 것: [있으면 명시]
- 이번 범위에서 제외할 것: [있으면 명시]
```

### 초안 작성 시 주의사항

1. **초안은 초안입니다** — 첫 버전이 완벽할 필요 없음
2. **빈 섹션은 괜찮습니다** — "이 부분은 추후 작성 예정"으로 표시
3. **모르는 건 Open Questions에** — 억지로 채우지 말고 질문으로 남기기

---

## Phase 4: 다각도 검토 (10분)

> **목적**: 내가 놓친 것을 찾아내기

이 단계가 PRD 품질을 가장 많이 높이는 단계입니다.

### 검토 방법: 역할 지정 검토

Claude에게 다른 팀원의 역할을 맡겨서 검토하게 합니다.

#### 개발자 관점 검토

```
방금 작성한 PRD를 시니어 백엔드 개발자 관점에서 검토해주세요.

다음을 찾아줘:
1. 구현하기 전에 PO에게 반드시 확인해야 할 것들
2. 기술적으로 구현이 어렵거나 비용이 클 수 있는 요구사항
3. 이 PRD만 봐서는 개발을 시작할 수 없는 불명확한 부분
4. 놓친 데이터 처리 케이스 (저장, 수정, 삭제, 오류 등)
```

#### UX 디자이너 관점 검토

```
방금 작성한 PRD를 시니어 UX 디자이너 관점에서 검토해주세요.

다음을 찾아줘:
1. 사용자가 혼란스러워할 수 있는 플로우
2. UX 흐름에서 빠진 화면이나 상태
3. 사용자가 실수할 수 있는 지점과 그 방지책
4. 접근성을 고려해야 할 부분
```

#### 사용자 관점 검토

```
방금 작성한 PRD를 읽고, 이 기능의 목표 사용자인 
[사용자 페르소나 설명] 입장에서 평가해주세요.

다음을 알려줘:
1. 이 기능을 쓰기 어렵거나 헷갈릴 부분
2. 기대했는데 없는 기능
3. 있긴 한데 쓸 것 같지 않은 기능
4. "왜 이게 이렇게 동작하지?"라고 생각할 순간들
```

---

## Phase 5: 최종 완성 (5분)

> **목적**: 검토 결과를 반영해서 팀 공유용 버전 만들기

### 최종화 프롬프트

```
방금 진행한 검토 내용을 반영해서 PRD를 최종 버전으로 업데이트해주세요.

추가로:
1. Open Questions 섹션을 검토하고 답변 가능한 것은 채워주세요
2. 각 Must Have 항목이 충분히 구체적인지 확인해주세요
   (개발팀이 바로 작업을 시작할 수 있는 수준인지)
3. 요약 섹션이 1-2문단 안에 핵심을 담고 있는지 확인해주세요
```

## 실전 PRD 작성 대화 예시

아래는 "결제 실패 재시도 유도 기능" PRD를 실제로 만드는 대화 흐름입니다.

### 1단계: 탐색

```
나: 커머스 앱에서 결제 실패 후 사용자가 이탈하는 문제가 있어요.
    '결제 실패 재시도 유도 기능'을 기획하려 합니다.
    
    이 기능에 대해 탐색해줘:
    - 결제 실패의 주요 원인 유형들
    - 각 원인에 따라 다른 UX가 필요한지
    - 대형 커머스에서 이 문제를 어떻게 처리하는지
    - 기획 시 반드시 챙겨야 할 케이스들

Claude: 결제 실패는 크게 3가지 유형으로 나뉩니다:
        1. 일시적 오류 (네트워크, 서버 오류) → 즉시 재시도 유도
        2. 카드 문제 (한도 초과, 정보 오류) → 다른 결제 수단 유도
        3. 사용자 취소 → 장바구니 유지 + 재방문 시 알림
        
        대형 커머스는 보통... (계속)
```

### 2단계: 구조 확정

```
나: 좋아. 이 기능 PRD에 필요한 섹션 구조를 제안해주세요.
    기본 구조에 추가로 필요한 섹션이 있으면 알려주세요.

Claude: 기본 구조에 추가로 다음 섹션을 권장합니다:
        - 결제 실패 유형 분류 (왜냐하면 유형마다 UX가 다르기 때문)
        - 결제 수단별 처리 방식 (카드사마다 오류 코드가 다름)
        - 알림/메시지 문구 가이드라인
```

### 3단계: 초안 작성

```
나: 이 구조로 PRD 초안 작성해주세요.
    우리 앱은 30대 여성 대상 패션 커머스고, 
    월 구매 전환율 3%를 5%로 높이는 게 Q1 목표입니다.

Claude: ## 결제 실패 재시도 유도 기능 PRD

        ### 0. 요약
        결제 실패 시 사용자 이탈을 줄이고 구매 완료율을 높이기 위한 
        실패 유형별 맞춤 재시도 유도 기능. Q1 목표인 전환율 5% 달성의 
        핵심 개선 항목 중 하나.
        ... (계속)
```

### 4단계: 개발자 관점 검토

```
나: 이 PRD를 백엔드 개발자 관점에서 검토해주세요.

Claude: 검토 결과, 다음 부분이 불명확합니다:
        1. 결제 실패 로그를 어느 기간 동안 보관할 것인가?
        2. 재시도 유도 알림은 실패 후 몇 분 뒤에 발송하는가?
        3. 카드 오류 코드는 카드사 API 응답 기준인가, 자체 분류인가?
        → Open Questions에 추가 권장
```

### 5단계: 최종화

```
나: 검토 결과 반영해서 최종 버전 완성해주세요.
    Open Questions에 방금 나온 3가지 추가해주세요.
```

## 자주 빠뜨리는 엣지케이스 체크리스트

PRD를 마치기 전에 아래 목록을 꼭 확인하세요. Claude에게 이 목록을 기준으로 검토를 요청할 수도 있습니다.

```
□ 로그인/비로그인 사용자를 다 고려했는가?
□ 네트워크 오류 시 어떻게 처리되는가?
□ 데이터가 없을 때 (empty state) 어떻게 보이는가?
□ 권한이 없는 사용자가 접근하면?
□ 동시에 여러 기기에서 접근하면?
□ 앱 업데이트 중간에 이 기능을 사용하면?
□ 사용자가 연속으로 같은 액션을 빠르게 하면?
□ 서버 응답이 매우 느릴 때 UX는?
□ 기능 사용 중 앱이 종료되면?
□ 최소 지원 OS/앱 버전 제한이 있는가?
```

## ✏️ 오늘의 실습 과제

**목표**: 실제 기능의 PRD를 40분 안에 완성하기

**선택**:
- A. 현재 진행 중인 실제 기능으로 시도
- B. 아래 연습 시나리오 사용

> **연습 시나리오**: 앱에서 "최근 검색어" 기능 추가  
> 사용자가 이전에 검색했던 키워드를 저장하고, 다시 검색할 때 빠르게 접근할 수 있는 기능

**체크리스트**:
- [ ] 5단계 플로우를 순서대로 진행했는가?
- [ ] 개발자 관점 검토를 했는가?
- [ ] 엣지케이스 체크리스트를 확인했는가?
- [ ] Open Questions가 명시되어 있는가?
- [ ] 40분 안에 완성했는가?

**완성 후**: `templates/PRD_template.md`를 채워서 저장해두세요.
