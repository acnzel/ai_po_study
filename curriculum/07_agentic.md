# 7강: 에이전트에게 일 위임하기
## — Agentic Workflow 입문

> **소요시간**: 45분  
> **이 강의를 마치면**: 에이전트 사고방식으로 PO 업무를 설계하고, 실제로 위임할 수 있다

---

## "일을 시키는 게 아니라, 일을 하는 시스템을 만드는 것"

에이전트 시스템을 제대로 활용하면 40일 만에 100만 줄짜리 소프트웨어를 만들 수 있습니다. 그 핵심은 이것입니다:

**코드를 직접 쓰지 않았습니다. 코드를 쓰는 에이전트 시스템을 만들었습니다.**

PO 입장으로 번역하면:

> **PRD를 직접 쓰는 게 아니라,  
>  PRD를 잘 쓰고 검토하고 개선하는 에이전트 시스템을 만드는 것**

---

## 에이전트가 뭔가요?

**에이전트(Agent)** = 특정 역할을 맡아서 작업을 수행하는 Claude

Claude 하나가 여러 역할을 순서대로, 또는 동시에 수행하게 만들 수 있습니다.

예를 들어:
- **탐색 에이전트**: "이 기능에 대해 조사해"
- **작성 에이전트**: "조사 내용으로 PRD 초안 써"
- **검토 에이전트**: "개발자 관점으로 이 PRD 비판해"
- **수정 에이전트**: "비판 내용 반영해서 개선해"

---

## Agentic Workflow의 핵심 구조

```
[나] ──── 방향 제시 & 최종 승인
  │
  ▼
[Soul Document] ──── 컨텍스트 & 규칙 제공
  │
  ▼
[에이전트 1: 탐색] ──── 정보 수집 & 분석
  │
  ▼
[에이전트 2: 초안 작성] ──── 결과물 생성
  │
  ▼
[에이전트 3: 검토] ──── 다각도 비판
  │
  ▼
[에이전트 4: 개선] ──── 피드백 반영
  │
  ▼
[나] ──── 검토 & 승인
```

핵심은: **내가 각 단계마다 세부 작업을 하는 게 아니라, 방향만 제시하고 최종 결과를 검토한다.**

---

## 핵심 기법: "비판을 스스로 시키기"

여러 각도에서의 비판을 Claude 스스로 하게 만드는 것, 이게 에이전트 활용의 핵심입니다.

이 기법이 강력한 이유: Claude가 자기 결과물을 스스로 비판할 때, **내가 보지 못한 문제**를 발견합니다.

### 비판 기법 실전

```
[PRD 초안 작성 후]

나: "방금 작성한 PRD에서 가장 큰 문제점 3가지를 찾아주세요.
     솔직하게, 혹독하게 비판해주세요."

Claude: [자기 결과물 비판]
        1. 엣지케이스 섹션이 너무 얕습니다. 
           결제 중 앱이 강제 종료되는 경우가 없음.
        2. 성공 지표가 구체적이지 않습니다. 
           "사용자 만족도 향상"은 측정 불가능.
        3. 개발팀 관점 설명이 부족합니다. 
           API 동작 방식이 불명확함.

나: "이 3가지를 모두 개선해주세요."
```

**결과**: 내가 직접 찾지 못했던 문제들이 수정된 고품질 PRD

## PO Agentic Workflow 실전 예시

### 시나리오: "장바구니 개선" 기능 기획

이 기능을 에이전트 방식으로 처리하면:

---

#### 에이전트 1: 탐색 에이전트

```
나: 장바구니 개선 기능을 기획할 예정이야.
    아래 역할로 탐색해주세요:
    
    역할: UX 리서처
    탐색 범위:
    - 커머스 앱 장바구니에서 사용자가 자주 겪는 불편함
    - 대형 커머스(쿠팡, 무신사 등)의 장바구니 UX 특징
    - 장바구니 이탈률을 낮추는 데 효과적인 기능들
    
    결과물: 기획 시 활용할 수 있는 인사이트 요약 (불릿 포인트)
```

---

#### 에이전트 2: 초안 작성 에이전트

```
나: 방금 탐색한 내용을 바탕으로
    "장바구니 경험 개선" PRD 초안을 작성해주세요.
    
    역할: PRD 작성 전문가
    제약사항:
    - 2주 스프린트 1회 분량
    - 개발자 4명, 디자이너 1명
    - Q1 목표: 구매 전환율 3% → 5%
```

---

#### 에이전트 3: 검토 에이전트

```
나: 위 PRD를 다음 3가지 관점에서 각각 검토해주세요.
    
    관점 1 - [iOS 개발자]:
    "이 PRD대로 개발하려면 뭐가 더 필요한가?"
    
    관점 2 - [30대 직장 여성 사용자]:
    "이 기능을 쓰다가 어디서 막힐 것 같은가?"
    
    관점 3 - [비즈니스 담당자]:
    "전환율 목표 달성에 이 기능이 충분한가?"
    
    각 관점마다 발견한 문제 3가지씩 알려주세요.
```

---

#### 에이전트 4: 개선 에이전트

```
나: 검토 결과를 반영해서 PRD를 최종 업데이트해주세요.
    
    변경사항 요약도 함께 작성해주세요:
    - 추가한 내용
    - 수정한 내용
    - 삭제한 내용
    - 여전히 미결인 Open Questions
```

---

### 전체 흐름 정리

| 단계 | 에이전트 역할 | 내가 하는 것 |
|------|-------------|------------|
| 탐색 | UX 리서처 | 결과 확인 & 방향 확정 |
| 초안 | PRD 작성가 | 결과 확인 & 추가 컨텍스트 제공 |
| 검토 | 개발자/사용자/비즈니스 | 결과 확인 & 우선순위 판단 |
| 개선 | 통합 편집자 | 최종 결과 검토 & 승인 |

**내가 실제로 쓴 시간**: 각 단계 전환 시 5분씩 = 총 20분  
**Claude가 작업한 시간**: 각 단계 5-10분씩 = 총 40분  
**결과물 품질**: 혼자 2시간 작업한 것과 동급 이상

---

## Agentic Workflow 사용 시 주의사항

### ⚠️ 주의 1: 최종 판단은 항상 내가 한다

Claude는 도구입니다. 아무리 잘 만들어진 에이전트 시스템이라도, 팀에 공유하기 전에는 반드시 내가 읽고 판단해야 합니다.

```
체크리스트: 에이전트 결과물 공유 전 확인
□ 사실이 아닌 내용이 포함되어 있지 않은가?
□ 우리 팀/서비스 상황에 맞는 내용인가?
□ 개인정보나 민감한 정보가 포함되어 있지 않은가?
□ 내가 동의할 수 없는 결정이 포함되어 있지 않은가?
```

### ⚠️ 주의 2: 처음부터 복잡하게 만들지 마라

에이전트 시스템은 단순하게 시작해서 점진적으로 발전시키세요.

```
단계 1 (지금): 에이전트 2-3개 순서대로 사용
단계 2 (1개월 후): 자주 쓰는 흐름을 Command로 저장
단계 3 (3개월 후): 여러 Command를 연결한 워크플로우
```

### ⚠️ 주의 3: 에이전트마다 역할을 명확히 하라

역할이 모호하면 에이전트가 방향을 잃습니다.

❌ 모호한 역할: "이 PRD를 검토해주세요"  
✅ 명확한 역할: "시니어 iOS 개발자 관점에서, 구현 전 결정이 필요한 사항만 찾아주세요"

## 에이전트 시스템의 진짜 가치

에이전트 시스템을 직접 활용해보면 발견하는 것:

사람이 피드백을 줄 때 실제 작업은 에이전트가 하는 구조가 되면, 기존에 병목이었던 것들이 사라집니다.

즉, **잘 설계된 에이전트 시스템은 병목이 사라집니다.**

PO 입장에서:
- 검토가 늦어서 기획이 지연되는 일이 없어짐
- 동일한 기능을 여러 관점에서 동시에 검토 가능
- 내가 쉬는 동안에도 초안 생성이 가능

---

## ✏️ 오늘의 실습 과제

**목표**: 실제 기능 하나를 에이전트 4단계 플로우로 처리하기

**Step 1**: 현재 기획해야 할 기능 1개 선택 (또는 4강 시나리오 계속 사용)

**Step 2**: 4단계 에이전트를 순서대로 실행

```
에이전트 1: 탐색 (역할: UX 리서처)
에이전트 2: 초안 (역할: PRD 작성 전문가)  
에이전트 3: 검토 (역할: 개발자 / 사용자 / 비즈니스)
에이전트 4: 개선 (역할: 통합 편집자)
```

**Step 3**: 각 단계 후 체크
- 결과물을 읽었는가?
- 방향을 수정해야 할 부분이 있었는가?
- 다음 에이전트에 전달할 핵심 메시지는?

**Step 4**: 최종 결과물 평가
- 혼자 했을 때 대비 시간은?
- 품질 차이는?
- 더 개선할 방법은?
