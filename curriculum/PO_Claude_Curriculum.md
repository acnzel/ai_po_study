## 1강: AI와 나는 어떤 관계인가?
### — 마인드셋 전환

### 핵심 메시지

> *"30분 배우는 노력을 한 다음에, 결국 본인이 3분 안에 처리할 수 있게 됐어요."*
> — 신정규 대표

Lablup의 CFO는 코딩을 전혀 모릅니다. 그런데 지금은 **Claude Code를 사용해서 업무를 직접 처리**하고 있어요. 마케터, 콘텐츠 담당자도 마찬가지입니다. 처음 30분을 투자해서 기초를 익히고 나면, **가속 곡선에 올라타게 됩니다.**

### AI와 협업하는 3가지 모드

```
❌ 잘못된 방식
  나: "PRD 써줘"
  Claude: (알아서 막 씀)
  나: "이게 아닌데…" (처음부터 다시)

⚠️ 보통 방식
  나: "이 기능의 PRD 써줘. 조건은 이렇고..."
  Claude: (나름 씀)
  나: "좋아, 그런데 이 부분은 고쳐줘"

✅ 잘하는 방식 (Agentic Workflow)
  나: "우리 서비스 컨텍스트를 먼저 이해해줘"
  Claude: "알겠습니다. 몇 가지 여쭤볼게요..."
  나: (대화로 컨텍스트 축적)
  나: "이제 PRD 초안 만들어줘"
  Claude: (맥락을 이해한 채로 작성)
  나: "비판적 시각으로 검토하고 보완해줘"
  Claude: (자기 검토 후 개선)
```

### 왜 이게 중요한가?

신정규 대표는 말합니다: **"내가 코딩을 하는 대상이 최종 결과물이 아니라, 코딩을 하는 애를 만드는 거예요."**

PO 입장으로 번역하면: **내가 문서를 쓰는 게 아니라, 문서를 잘 쓰는 시스템을 만드는 것.**

### 실습 과제
- Claude에게 오늘 내 주요 업무 3가지를 말해보기
- "이 중에서 어떤 걸 먼저 도와드릴까요?"라는 답이 오는지 확인
- 느낌 기록하기: 어색한 점, 생각보다 잘 된 점

## 2강: Claude를 처음 만나는 법
### — 기초 프롬프팅

### 황금 원칙: 한 번에 다 하려 하지 말 것

> *"원하는 걸 한 번에 들어가지 않는 게 훨씬 더 좋은 결과를 가져오더라고요."*
> — 신정규 대표

#### BAD: 한 방에 다 요구
```
"우리 앱의 알림 설정 기능 PRD를 작성해줘. 
사용자 분석은 MZ세대 중심이고, 
기술 스택은 React Native고, 
출시일은 3월이야. 다 써줘."
```

#### GOOD: 단계적 컨텍스트 쌓기
```
Step 1: "우리 앱에 대해 설명할게. 
         [앱 설명] — 이 앱에 대해 이해한 내용 요약해줘."

Step 2: "알림 설정 기능을 추가하려 해. 
         고려해야 할 것들이 무엇이 있을까?"

Step 3: "좋아. 이제 이 기능의 PRD 초안을 작성해줘."

Step 4: "작성한 PRD를 PO, 개발자, 디자이너 각각의 
         관점에서 비판적으로 검토해줘."
```

### 실전 프롬프팅 패턴 5가지

#### 패턴 1: 탐색 먼저
```
"[주제]에 대해 탐색하고, 어떤 내용을 다루는지 알려줘.
 그리고 내가 고려해야 할 점들이 무엇인지 정리해줘."
```

#### 패턴 2: 구조 제안 요청
```
"[문서 유형]을 작성할 때 담아야 할 섹션과 순서를 제안해줘.
 직접 쓰지는 말고, 구조만 먼저."
```

#### 패턴 3: 비판적 검토
```
"방금 작성한 내용을 [역할] 관점에서 비판적으로 검토해줘.
 무엇이 빠졌고, 무엇이 불명확한지 알려줘."
```

#### 패턴 4: 반복 개선
```
"위 피드백을 반영해서 개선된 버전을 만들어줘.
 변경한 부분과 이유도 함께 설명해줘."
```

#### 패턴 5: 관점 전환
```
"같은 내용을 [새로운 독자]가 읽는다고 가정하고 다시 써줘."
```

### 언어 팁

신정규 대표는 영어 대신 **한국어로 입력**하고, **존댓말**을 사용합니다.

- 한국어 품질이 충분히 좋아짐
- 한국어로 타이핑하는 게 더 빠름
- 존댓말은 본인 습관 보호 목적도 있음 (AI에게 반말하다가 사람에게도...)

### 실습 과제
위의 5가지 패턴을 각각 한 번씩 써보며 PRD 섹션 하나 완성하기

## 3강: Soul Document 만들기
### — 나만의 CLAUDE.md 작성

### Soul Document란?

> *"CLAUDE.md라는 일종의 soul document 하나가 생겨요.
>  Claude Code를 쓰든 Claude Co-work를 쓰든 
>  이 폴더에서 시작하게 되면 항상 맨 처음 읽는 파일입니다."*
> — 신정규 대표

Claude는 매 대화를 새로 시작합니다. **Soul Document는 Claude에게 "나는 누구고, 우리 프로젝트는 무엇이며, 어떻게 행동해야 하는지"를 알려주는 파일**입니다.

### PO를 위한 Soul Document 구조

```markdown
# [프로젝트명] - Claude 작업 가이드

## 나(PO)에 대해
- 이름/역할: [시니어 PO, 커머스 도메인 5년]
- 우리 제품: [간략한 서비스 설명]
- 현재 단계: [베타 서비스 운영 중 / 0→1 중 / 스케일업 중]

## 우리 제품에 대해
- 핵심 사용자: [누구]
- 해결하는 문제: [무엇]
- 경쟁 제품 대비 차별점: [무엇]
- 현재 주요 KPI: [무엇]

## 작업 방식
- 문서 작성 시 항상 사용자 관점 먼저
- PRD 초안 후 반드시 개발자/디자이너 관점 검토 추가
- 불명확한 부분은 작성 전 질문으로 확인
- 한국어로 작성

## 자주 쓰는 문서 형식
- PRD 템플릿: [링크 또는 간략 구조]
- 회의록 양식: [링크 또는 간략 구조]

## 현재 진행 중인 프로젝트
- [프로젝트 A]: [간략 설명]
- [프로젝트 B]: [간략 설명]
```

### Soul Document를 만드는 방법

직접 작성하기 어렵다면, **Claude에게 대화로 만들어 달라고 요청**하면 됩니다:

```
"나는 [회사명]의 PO야. 우리 서비스는 [설명]이고, 
 내가 주로 작성하는 문서는 PRD, 기획서, 스프린트 리뷰 등이야.
 이 정보를 바탕으로 나와 앞으로 잘 협업하기 위한 
 CLAUDE.md 초안을 작성해줘. 
 추가로 필요한 정보가 있으면 먼저 질문해줘."
```

### Soul Document의 힘

컨텍스트 없이 작업 요청할 때:
```
나: "결제 실패 알림 기능 PRD 써줘"
Claude: (일반적인 양식으로 작성)
```

Soul Document가 있을 때:
```
나: "결제 실패 알림 기능 PRD 써줘"
Claude: (우리 서비스 특성, 사용자 페르소나, 
         기존 문서 스타일을 반영해서 작성)
```

### 실습 과제
자신의 프로젝트용 Soul Document 초안 작성 (Claude와 함께)

## 4강: PRD를 Claude와 함께 쓰는 법
### — PRD 작성 실습

### PRD 작성을 망치는 흔한 실수

1. **결과물부터 요청** — 맥락 없이 "PRD 써줘"
2. **한 번에 완성하려는 욕심** — 초안에서 바로 완성본을 기대
3. **검토를 건너뜀** — 작성 후 비판적 검토 없이 공유
4. **엣지케이스 누락** — 예외 상황을 고려하지 않음

### 추천 PRD 작성 플로우

```
Phase 1: 탐색 (10분)
├── 기능 배경과 목적 정리
├── 관련 사용자 시나리오 탐색
└── 고려해야 할 요소 목록화

Phase 2: 구조 설계 (5분)
├── PRD 섹션 구조 확정
└── 각 섹션별 작성 포인트 정리

Phase 3: 초안 작성 (10분)
└── 섹션별로 순차 작성

Phase 4: 다각도 검토 (10분)
├── 사용자 관점 검토
├── 개발자 관점 검토
├── 디자이너 관점 검토
└── 엣지케이스 보완

Phase 5: 최종 완성 (5분)
└── 피드백 반영 및 정리
```

### PO 표준 PRD 템플릿 (Claude 활용 버전)

아래 구조를 Claude에게 주고 각 섹션을 채워달라고 요청하세요:

```markdown
## [기능명] PRD

### 1. 배경 및 목적
- 이 기능이 왜 필요한가?
- 어떤 사용자 문제를 해결하는가?
- 비즈니스 목표와의 연결

### 2. 목표 사용자 & 시나리오
- 주요 사용자 페르소나
- 핵심 사용 시나리오 (As-Is → To-Be)

### 3. 기능 요구사항
- 필수 기능 (Must Have)
- 선택 기능 (Nice to Have)
- 제외 범위 (Out of Scope)

### 4. UX 흐름
- 주요 플로우 설명
- 화면별 동작 정의

### 5. 엣지케이스 & 예외 처리
- 예외 상황 목록
- 각 예외에 대한 처리 방안

### 6. 성공 지표
- 이 기능의 성공을 어떻게 측정하는가?

### 7. 미결 사항 (Open Questions)
- 아직 결정되지 않은 것들
```

### Claude 활용 PRD 작성 실전 대화 예시

```
[1단계 - 배경 탐색]
나: "우리 앱에서 사용자들이 결제 실패 후 이탈하는 문제가 있어.
     이 문제를 해결하기 위한 '결제 실패 재시도 유도 기능'을 
     기획하려 해. 이 기능에 대해 고려해야 할 것들이 뭐가 있을지 탐색해줘."

[2단계 - 사용자 시나리오]
나: "좋아. 이제 이 기능을 사용할 사용자 시나리오를 
     3가지 케이스로 나눠서 작성해줘. 
     '일시적 네트워크 오류', '카드 한도 초과', '카드 정보 오류' 케이스로."

[3단계 - PRD 초안]
나: "위 내용을 바탕으로 PRD 초안을 작성해줘. 
     위에 있는 PRD 템플릿 구조로."

[4단계 - 검토]
나: "작성된 PRD를 개발자 관점에서 검토해줘. 
     기술 구현 시 불명확한 부분이나 추가 논의가 필요한 부분을 찾아줘."

[5단계 - 보완]
나: "엣지케이스 섹션이 약한 것 같아. 
     결제 시스템에서 발생할 수 있는 엣지케이스를 더 보완해줘."
```

## 5강: 컨텍스트 쌓기
### — 질문 먼저, 작업 나중에

### 신정규 대표의 핵심 인사이트

> *"내가 뭘 하고 싶은지가 명확하지만 
>  그건 내 머릿속에만 명확하기 때문에 
>  Claude의 컨텍스트 메모리에 집어넣기 위해서 
>  사전 조사를 시키는 거예요."*

### 컨텍스트 쌓기 3단계

```
Step 1: 배경 조사 시키기
→ "이 주제에 대해 탐색하고 핵심을 정리해줘"
→ 목적: Claude가 내 도메인을 이해하게 만들기

Step 2: 제약 조건 명확화
→ "우리 상황에서 고려해야 할 제약들이 뭐가 있을까?"
→ 목적: 현실적인 해결책 도출

Step 3: 작업 실행
→ "이제 이걸 바탕으로 [문서명]을 작성해줘"
→ 목적: 맥락에 맞는 결과물 생성
```

### 실전 예시: 스프린트 계획 미팅 준비

```
[배경 조사]
나: "다음 스프린트에서 '소셜 로그인 기능'을 출시할 예정이야.
     이 기능 출시 전 PO가 확인해야 할 체크리스트를 만들어줘."

[제약 조건 명확화]
나: "우리는 스타트업이고, 개발팀이 4명이야. 
     출시까지 2주 남았어. 이 상황에서 
     반드시 챙겨야 할 것과 타협 가능한 것을 나눠서 정리해줘."

[미팅 자료 생성]
나: "이 내용을 바탕으로 30분짜리 스프린트 계획 미팅 
     아젠다와 논의 포인트를 작성해줘."
```

### 컨텍스트가 쌓이면 달라지는 것

같은 요청이라도 컨텍스트 양에 따라 결과물이 달라집니다:

| 컨텍스트 없음 | 컨텍스트 풍부 |
|--------------|--------------|
| 일반적인 PRD 형식 | 우리 서비스에 맞는 PRD |
| 교과서적인 해결책 | 우리 팀 규모에 맞는 해결책 |
| 보편적인 엣지케이스 | 우리 사용자가 실제 겪을 엣지케이스 |

### PROGRESS.md - 진행 상황 추적

긴 작업을 할 때는 중간 상황을 기록해두면 다음 세션에도 이어받을 수 있습니다:

```markdown
# PROGRESS.md

## 현재 작업: [기능명] PRD
- [x] 배경 및 목적 초안 완료
- [x] 사용자 시나리오 3개 작성
- [ ] 기능 요구사항 정리 (진행 중)
- [ ] 엣지케이스 검토
- [ ] 개발팀 검토 반영

## 다음 작업 시 참고사항
- 결제 모듈 연동 방식에 대해 개발팀 확인 필요
- 디자이너와 UX 플로우 협의 예정
```

### 실습 과제
현재 기획 중인 프로젝트 하나에 대해 Claude와 20분 컨텍스트 쌓기 대화 진행 후 PROGRESS.md 작성

## 6강: 반복 업무 자동화

### 내 일이 줄어드는 게 목표

>  내 걸 만들게 되면 그 가속이 시작이 됩니다. 
>  왜냐하면 내 일이 줄어드는 게 핵심이 돼야 되거든요."*
> — 신정규 대표

### PO가 자주 하는 반복 업무들

- 스프린트 리뷰 회의록 작성
- 기능 요청 티켓 정리
- 주간 업무 보고 초안 작성
- 경쟁사 기능 비교 분석
- 사용자 피드백 분류 및 요약

### Claude Command/Skill 만드는 법

Claude Code나 Claude.ai Projects에서 자주 쓰는 작업을 **명령어**로 등록할 수 있습니다.

#### 예시: 스프린트 리뷰 자동화 Command

```markdown
# /sprint-review

## 역할
스프린트 리뷰 회의록과 다음 스프린트 준비 자료를 작성한다.

## 입력값
- 이번 스프린트에서 완료된 작업 목록
- 완료하지 못한 작업과 이유
- 주요 이슈 및 학습점

## 출력물
1. 스프린트 리뷰 회의록 (슬랙 공유용)
2. 다음 스프린트 개선 포인트 3가지
3. 팀에 공유할 주요 성과 요약 (1-2문장)

## 형식
- 한국어, 간결하게
- 불릿 포인트 위주
- 기술 용어는 풀어서 설명
```

#### 예시: 기능 요청 분석 Command

```markdown
# /feature-request-analysis

## 역할
사용자로부터 들어온 기능 요청을 분석하고 우선순위를 제안한다.

## 입력값
- 기능 요청 내용 (raw text)
- 현재 로드맵 컨텍스트 (있으면)

## 출력물
1. 요청 재정의: 사용자가 진짜 원하는 게 무엇인가?
2. 영향도 분석: 구현 시 어떤 사용자에게 얼마나 영향?
3. 유사 기능 존재 여부: 이미 있는 기능으로 해결 가능한가?
4. 우선순위 제안: High/Medium/Low 및 근거
5. 다음 액션: PO가 취해야 할 다음 단계
```

3. 자주 쓰는 작업은 Instructions에 "명령어 형태"로 등록
4. 이후 대화 시 "스프린트 리뷰 해줘" 한 마디로 자동 실행

## 7강: 에이전트에게 일 위임하기
### — Agentic Workflow 입문

### 에이전트 사고방식으로의 전환

> *"내가 코딩을 하는 대상이 최종 결과물이 아니라, 
>  코딩을 하는 애를 만드는 거예요."*

PO 버전으로 번역: **내가 문서를 쓰는 게 아니라, 문서를 잘 써주는 에이전트 시스템을 만드는 것**

### Agentic Workflow의 핵심 구조

```
[나] (방향 제시 & 검토)
  ↓
[Soul Document] (컨텍스트 & 규칙)
  ↓
[에이전트] (실제 작업 수행)
  ↓
[결과물] (PRD, 회의록, 분석 보고서 등)
  ↓
[나] (승인 or 피드백)
  ↓
[에이전트] (개선)
```

### PO Agentic Workflow 실전 예시

#### 시나리오: 기능 기획 전체 프로세스 자동화

```
Step 1: 탐색 에이전트
"[기능명]에 대해 경쟁사 사례, 사용자 니즈, 기술적 고려사항을
 조사해서 briefing document를 만들어줘."

Step 2: 초안 에이전트  
"briefing document를 기반으로 PRD 초안을 작성해줘."

Step 3: 검토 에이전트
"작성된 PRD를 다음 3가지 관점에서 검토해줘:
 (1) 사용자 관점: 사용성 문제가 있는가?
 (2) 개발 관점: 구현 불명확한 부분이 있는가?  
 (3) 비즈니스 관점: 목표와 잘 연결되는가?"

Step 4: 최종 정리 에이전트
"검토 결과를 반영해서 최종 PRD를 작성하고,
 팀 공유용 요약 버전도 함께 만들어줘."
```

### "비판을 스스로 시키기" 기법

신정규 대표의 핵심 기법: **에이전트에게 자기 결과물을 스스로 비판하게 만들기**

```
나: "[PRD 내용]"

Claude: (PRD 작성 완료)

나: "방금 작성한 PRD를 다음 관점에서 비판해줘:
     - 어떤 부분이 가장 불명확한가?
     - 개발팀이 가장 많이 질문할 부분은?
     - 사용자가 혼란스러워할 UX 포인트는?
     - 빠진 엣지케이스는?"

Claude: (자기 작성물 비판)

나: "비판 내용을 반영해서 개선해줘."
```

### 에이전트 활용 주의사항

1. **최종 판단은 항상 사람이 한다** — Claude는 초안과 검토를 돕는 것
2. **구체적인 지시가 좋은 결과를 만든다** — 막연하면 막연한 결과가 나옴
3. **컨텍스트 유지가 중요** — 긴 작업은 PROGRESS.md로 중간 저장
4. **단계를 건너뛰지 않는다** — 급하더라도 탐색 → 초안 → 검토 순서 지키기

### 실습 과제
진행 중인 프로젝트에서 다음 주에 기획해야 할 기능 하나를 골라서 위의 4단계 에이전트 플로우로 PRD 완성하기

## 8강: 내 Workflow 완성하기
### — 종합 실습

### 나만의 PO-Claude Workflow 설계

이제까지 배운 것들을 종합해서 **나만의 업무 workflow**를 완성합니다.

#### Workflow 설계 템플릿

```
📋 나의 PO-Claude Workflow

[프로젝트 시작 시]
□ Soul Document (CLAUDE.md) 업데이트
□ PROGRESS.md에 목표와 일정 기록
□ 배경 탐색 대화 진행

[기능 기획 시]
□ Phase 1: 탐색 (경쟁사, 사용자 니즈, 기술)
□ Phase 2: PRD 구조 설계
□ Phase 3: 섹션별 초안 작성
□ Phase 4: 다각도 검토 (사용자/개발/비즈니스)
□ Phase 5: 팀 공유용 최종 버전

[반복 업무]
□ 스프린트 리뷰 → /sprint-review 명령어
□ 이해관계자 보고 → /stakeholder-update
□ 기능 요청 분석 → /feature-analysis

[주기적으로]
□ Soul Document 업데이트 (새로운 컨텍스트 추가)
□ Command 개선 (효과적이지 않은 것 수정)
□ 학습 내용 기록 (잘 된 것/안 된 것)
```

### 최종 체크리스트: 초보 PO를 위한 Claude 활용 10계명

```
✅ 1. Soul Document를 만들고 꾸준히 업데이트한다
✅ 2. 한 번에 다 요구하지 않는다 — 탐색 먼저
✅ 3. 결과물 생성 전 충분한 컨텍스트를 쌓는다
✅ 4. 초안이 나오면 바로 다각도 검토를 요청한다
✅ 5. 에이전트에게 자기 결과물을 비판하게 만든다
✅ 6. 반복 업무는 Command로 만들어 자동화한다
✅ 7. 긴 작업은 PROGRESS.md로 중간 저장한다
✅ 8. 최종 판단은 항상 내가 한다
✅ 9. 효과적인 프롬프트는 기록해두고 재활용한다
✅ 10. 안 되는 게 있으면 2주 후에 다시 시도한다
```

### 실습 과제 (Final Project)

다음 조건을 만족하는 **실제 PRD 1개** 완성하기:

```
요구사항:
□ 자신의 현재 프로젝트에서 가장 중요한 기능 선택
□ Soul Document 기반으로 작업
□ 탐색 → 구조 → 초안 → 검토 → 최종 플로우 준수
□ 다각도 검토 (사용자/개발/비즈니스) 완료
□ 엣지케이스 섹션 포함
□ 총 소요 시간 1시간 이내 목표

제출물:
1. 완성된 PRD
2. 작업 중 사용한 프롬프트 3-5개 (효과적이었던 것)
3. 학습 회고 (잘 된 점, 어려웠던 점, 개선할 점)
```

```
# 탐색

"이 기능의 사용자 시나리오를 [n]가지로 나눠서 작성해줘.

"이 PRD에서 빠진 엣지케이스를 찾아줘. 
 특히 [에러 상황], [권한 이슈], [데이터 누락] 측면에서."

# 개발자 관점 검토
"이 PRD를 개발자가 읽는다고 가정하고, 
 구현이 불명확하거나 추가 결정이 필요한 부분을 모두 찾아줘."
```

### 회의 관련

```
# 회의 준비
"[회의 목적]을 위한 1시간 회의 아젠다를 만들어줘. 
 각 아젠다는 소요 시간, 논의 포인트, 결정해야 할 것을 포함해줘."

# 회의록 작성
"다음 내용을 회의록으로 정리해줘.
```
