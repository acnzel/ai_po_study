# 6강: 반복 업무 자동화
## — Command 만들기

> **소요시간**: 60분  
> **이 강의를 마치면**: 자주 하는 반복 업무 1가지를 Command로 만들어서 실제로 사용할 수 있다

---

## "내 일이 줄어드는 게 목표입니다"

> *"기본적으로 이 가속 구간에 올라타려면  
>  내 걸 만들게 되면 그 가속이 시작됩니다.  
>  왜냐하면 내 일이 줄어드는 게 핵심이 돼야 되거든요."*  
> — 신정규 (Lablup 대표)

이 강의의 핵심은 하나입니다: **남이 만든 걸 쓰는 게 아니라, 나만의 도구를 만들어라.**

남이 만든 프롬프트 100개를 아는 것보다, **내 업무에 딱 맞는 Command 3개를 직접 만드는 것**이 훨씬 강력합니다.

---

## Command가 뭔가요?

**Command = 자주 하는 작업을 명령어 하나로 실행하게 만든 나만의 도구**

예를 들어, 매주 스프린트 리뷰를 해야 한다면:

### Command 없이:
```
나: "이번 스프린트 리뷰 회의록 써줘. 우리 팀은 2주 스프린트고, 
     완료된 작업은 이거고, 못 한 건 이거야. 
     형식은 슬랙에 붙여넣을 수 있게, 기술 용어는 풀어서..."
     
→ 매주 이 설명을 반복
```

### Command 있을 때:
```
나: "스프린트 리뷰 해줘. 완료: [목록], 미완: [목록]"

→ Command 안에 모든 설정이 담겨있어서 즉시 실행
```

---

## PO가 자동화하기 좋은 반복 업무

아래 목록 중 **매주 2번 이상** 하는 것을 찾아보세요. 그게 Command로 만들 후보입니다.

### 문서 작성 반복 업무
- [ ] 스프린트 리뷰 회의록
- [ ] 이해관계자 보고 요약
- [ ] 릴리즈 노트
- [ ] 주간 업무 보고
- [ ] 사용자 인터뷰 요약

### 분석 반복 업무
- [ ] 기능 요청 티켓 분석 및 우선순위
- [ ] 사용자 피드백 분류
- [ ] 경쟁사 기능 비교
- [ ] 데이터 인사이트 요약

### 기획 반복 업무
- [ ] PRD 초안 빠르게 생성
- [ ] 유저 스토리 작성
- [ ] 회고 (Retrospective) 진행
- [ ] 스프린트 계획 아젠다

## Command를 만드는 방법

Command는 사실 **잘 만든 프롬프트 문서**입니다. 특별한 기술이 필요 없어요.

### Command 명세서 구조

```markdown
# [Command 이름]

## 목적 (한 줄)
[이 Command가 하는 일]

## 언제 쓰나요?
[어떤 상황에서 이 Command를 실행하나요]

## 입력 (내가 제공하는 정보)
- [필수] [정보 이름]: [설명]
- [필수] [정보 이름]: [설명]
- [선택] [정보 이름]: [설명]

## 실행 프롬프트
[Claude에게 보내는 실제 프롬프트. {입력값} 형태로 변수 표시]

## 출력 형식
[어떤 형태로 결과가 나와야 하는가]

## 예시 입력/출력
[실제 사용 예시]
```

---

## 실전 Command 예시 3가지

### Command 1: 스프린트 리뷰

```markdown
# /sprint-review

## 목적
스프린트 종료 시 회의록 및 슬랙 공유용 요약을 즉시 작성

## 언제 쓰나요?
매 스프린트 마지막 날 또는 리뷰 미팅 후

## 입력
- [필수] 완료된 작업 목록
- [필수] 완료하지 못한 작업과 이유
- [선택] 주요 이슈나 배운 점

## 실행 프롬프트
다음 내용으로 스프린트 리뷰 회의록을 작성해줘.

완료 작업: {완료_목록}
미완 작업: {미완_목록}
특이사항: {특이사항}

작성 형식:
1. 슬랙 공유용 요약 (5줄 이내, 이모지 포함, 팀 전체 이해 가능한 언어)
2. 미완 작업 원인 분석 및 다음 스프린트 반영 사항
3. 다음 스프린트 시작 전 결정해야 할 것들

기술 용어는 비개발자도 이해할 수 있게 풀어서 써줘.

## 출력 형식
- 슬랙 채널에 바로 붙여넣을 수 있는 형태
- 기술 용어 없이 팀 전체가 이해 가능한 언어
```

**실제 사용:**
```
/sprint-review 실행해줘.

완료: 결제 실패 알림 기능, 마이페이지 UI 개선
미완: 소셜 로그인 (카카오 API 연동 이슈로 다음 스프린트 이월)
특이사항: 이번 스프린트에 버그 리포트가 평소보다 30% 많았음
```

---

### Command 2: 기능 요청 분석

```markdown
# /feature-analysis

## 목적
사용자 또는 내부에서 들어온 기능 요청을 빠르게 분석하고 의사결정 자료 생성

## 언제 쓰나요?
CS팀 피드백, 사용자 인터뷰, Jira 티켓 등 기능 요청이 들어왔을 때

## 입력
- [필수] 기능 요청 내용 (원문 그대로)
- [선택] 요청한 사람/채널 (사용자 유형 파악용)
- [선택] 현재 로드맵 컨텍스트

## 실행 프롬프트
아래 기능 요청을 분석해줘.

요청 내용: {요청_내용}
요청 출처: {출처}
현재 로드맵: {로드맵}

분석 항목:
1. 요청 재정의: 사용자가 진짜 원하는 게 무엇인가? (표면 요청 vs 근본 니즈)
2. 대상 사용자: 이 요청이 도움되는 사용자는 전체의 몇 %일까?
3. 기존 해결법: 현재 기능으로 해결 가능한가? 우회 방법은?
4. 구현 복잡도 예상: Low/Medium/High
5. 우선순위 제안: High/Medium/Low, 이유 포함
6. 다음 액션: PO가 취해야 할 다음 단계

## 출력 형식
구조화된 분석 보고서, Jira 티켓에 바로 붙여넣을 수 있는 형태
```

**실제 사용:**
```
/feature-analysis 실행해줘.

요청 내용: "앱에서 상품 여러 개를 동시에 공유할 수 있으면 좋겠어요. 
             지금은 하나씩밖에 못해서 친구한테 추천할 때 불편해요."
요청 출처: 앱스토어 리뷰 (별점 3점)
```

### Command 3: PRD 초안 빠르게 생성

```markdown
# /prd-draft

## 목적
기능 아이디어를 5분 안에 PRD 초안으로 변환

## 언제 쓰나요?
새 기능 아이디어가 생겼을 때, 빠르게 팀 공유용 초안이 필요할 때

## 입력
- [필수] 기능 이름
- [필수] 기능 한 줄 설명
- [필수] 해결하려는 사용자 문제
- [선택] 알고 있는 제약사항
- [선택] 참고 사례

## 실행 프롬프트
아래 정보로 PRD 초안을 빠르게 작성해줘.
완성도보다 빠른 공유가 목적이야.

기능명: {기능명}
설명: {설명}
해결 문제: {문제}
제약사항: {제약사항}
참고 사례: {참고}

포함할 섹션:
- 요약 (3줄 이내)
- 배경 및 목적
- 주요 기능 요구사항 (Must Have만, 5개 이내)
- 예상 사용자 시나리오 (1개)
- Open Questions (3개 이상)

마지막에: "이 초안의 가장 불명확한 부분 2가지를 알려줘"

## 출력 형식
팀 채팅에 바로 공유 가능한 간결한 형태 (A4 1페이지 이내)
```

---

## Claude.ai Projects에 Command 등록하기

### 방법 1: Project Instructions에 등록 (권장)

1. Claude.ai → Projects → [프로젝트 선택] → Instructions 탭
2. Soul Document 아래에 Command들을 추가

```
[Soul Document 내용]

---

## Commands

### /sprint-review
[Command 내용]

### /feature-analysis
[Command 내용]

### /prd-draft
[Command 내용]
```

3. 이제 대화에서 "/sprint-review 실행해줘"라고 하면 자동으로 실행

### 방법 2: 텍스트 파일로 저장해서 붙여넣기

Command들을 파일로 저장해두고, 필요할 때 복사해서 붙여넣습니다.

```
📁 my-commands/
  sprint-review.md
  feature-analysis.md
  prd-draft.md
  weekly-report.md
```

## 내 Command를 발전시키는 법

Command를 만들고 나면 끝이 아닙니다. 쓰면서 계속 개선해야 합니다.

### 개선 사이클

```
1. Command 사용
↓
2. 결과물 확인: "이 부분이 항상 애매하게 나오네"
↓
3. Command 프롬프트 수정: 불명확한 부분 구체화
↓
4. 다시 사용 → 더 좋은 결과
```

### Claude에게 Command 개선을 맡기기

```
내가 쓰는 /sprint-review Command인데, 
결과물에서 매번 이 부분이 아쉬워:
[아쉬운 점 설명]

이 Command 프롬프트를 어떻게 개선하면 좋을까?
```

## ✏️ 오늘의 실습 과제

**목표**: 나만의 Command 1개 만들고 실제 사용해보기

**Step 1**: 자주 하는 반복 업무 1가지 선택  
(스프린트 리뷰, 기능 분석, 회의록 등)

**Step 2**: 위의 Command 명세서 구조로 나만의 Command 작성

**Step 3**: Claude에게 다음을 요청:
```
내가 만든 Command야:
[Command 내용 붙여넣기]

이 Command를 개선할 수 있는 점이 있으면 알려줘.
특히 입력이 불명확하거나, 출력 형식이 너무 막연한 부분을.
```

**Step 4**: 개선 후 실제 업무에 적용해서 테스트

**Step 5**: 결과 평가:
- 기대한 결과가 나왔나?
- 수정이 얼마나 필요했나?
- 개선할 부분은?

## 나만의 PO Toolkit 구성 예시

최종적으로 이런 Command Set을 만들어가면 됩니다:

```
📁 PO Claude Toolkit

## 기획 도구
/prd-draft         → PRD 초안 5분 완성
/user-story        → 유저 스토리 자동 생성
/feature-analysis  → 기능 요청 빠른 분석

## 회의 도구  
/meeting-prep      → 회의 아젠다 준비
/meeting-notes     → 회의록 빠른 작성
/sprint-review     → 스프린트 리뷰 자동화
/retrospective     → 회고 진행 가이드

## 커뮤니케이션 도구
/stakeholder-update → 이해관계자 보고 요약
/release-note      → 릴리즈 노트 작성
/slack-summary     → 슬랙 공유용 요약
```

처음부터 이걸 다 만들려 하지 마세요. **지금 당장 가장 자주 하는 것 1개**부터 시작하면 됩니다.
